# 理论备忘：SAIR 等式蕴含竞赛的问题本质、算法视角与 Prompt 路线的关系

## 1. 一句话结论

这个竞赛对应的并不是“只能靠大模型黑箱猜”的问题，而是一个标准的**等式逻辑 / 普遍代数中的决策问题**：  
给定 `Equation 1` 与 `Equation 2`，判断是否对所有 magma（只带一个二元运算的代数结构）都有

`Equation 1 成立  =>  Equation 2 成立`

也就是判断 `Equation 1` 是否**蕴含** `Equation 2`。

但这个问题**不能简单粗暴地概括成“它到底是 P 还是 NP”**。更准确的说法是：

- 它当然可以做算法理论分析；
- 不同限制条件下，其可判定性和复杂度会显著变化；
- 一般的 equational consequence / equational theory 问题并不统一落在一个简单复杂度类里；
- 本竞赛之所以采用 prompt + 黑箱模型的形式，更多是**竞赛接口设计与蒸馏目标的约束**，而不是因为这个数学问题没有符号算法或形式化方法。

---

## 2. 这个竞赛问题在数学上到底是什么

### 2.1 对象：magma 与等式

竞赛里的对象是 **magma**：一个集合配上一个二元运算 `*`。  
题目中的表达式如

- `x = y * (z * w)`
- `x * y = (z * w) * u`

都可以看成是这种二元运算语言中的**项（term）**与**等式（identity）**。

### 2.2 任务：判断等式蕴含

题目要求判断：

- 是否任意一个满足 `Equation 1` 的 magma，
- 也一定满足 `Equation 2`。

这就是一个典型的**语义蕴含问题**，可写成：

`Equation 1 |= Equation 2`

从语义上看，它和“对所有模型是否成立”有关；  
从证明论上看，它和“能否从前提等式推导出目标等式”有关。

### 2.3 语义视角、证明视角、反例视角其实是同一件事的三种说法

对于本项目，最重要的是把这三种视角统一起来理解：

1. **语义视角**
   - 问的是：所有满足前提的结构里，目标是否都成立。

2. **证明视角**
   - 问的是：目标等式能否由前提等式推出。

3. **反例视角**
   - 若能构造一个满足 `Equation 1` 但不满足 `Equation 2` 的 magma，那么答案就是 `false`。

McNulty 的《Equational Logic》给出了等式逻辑的完备性表述：  
`Σ |= s ≈ t` 当且仅当 `Σ ⊢ s ≈ t`，也就是“语义蕴含”与“可推导性”是一致的。  
这说明该类问题天然就可以被看作**形式推导问题**，而不是纯经验猜测问题。  

---

## 3. 算法视角：它完全可以被当作一个决策问题

### 3.1 输入、输出都很清楚

从算法角度，本问题可以形式化成：

- **输入**：两个等式 `E1, E2`
- **输出**：布尔值  
  - `true`：若 `E1 |= E2`
  - `false`：若 `E1` 不蕴含 `E2`

输入可以用字符串表示，也可以用语法树表示。  
如果进一步工程化，复杂度通常会和这些量相关：

- 项的节点数
- 运算符深度
- 变量个数
- 左右两式是否共享变量
- 是否引入新变量
- 是否允许多个前提而非单前提

### 3.2 可行的算法性求解思路

从理论与系统角度，至少有四条典型路线：

1. **证明搜索 / 重写系统**
   - 试图把前提转成可用的重写规则、规范形或等式推导过程。

2. **反例搜索 / 有限模型搜索**
   - 试图找到一个有限 magma，使得 `E1` 成立但 `E2` 不成立。

3. **约束求解 / SMT / ATP**
   - 把“是否存在反例”转成约束问题交给求解器。

4. **图结构或统计预测**
   - 把等式对视为结构对象，用 ML 预测哪些边更可能为真，但这更接近启发式排序，而不是最终证明。

Equational Theories Project（ETP）本身就同时使用了：

- Lean 形式化验证
- 自动定理证明器
- 有限 magma 探索器
- Z3 反例生成
- implication graph 可视化

这恰恰说明：**这个问题非常适合算法与形式化工具介入**。

## 4. 为什么不能简单说“它是 P 还是 NP”

### 4.1 “P/NP”只覆盖一部分复杂度讨论

很多人一听“判定题”就会问：它是 `P` 还是 `NP`？  
这个问题并不是完全错，但对这里来说往往**不够准确**，原因是：

- `P/NP` 讨论的前提是：问题已经是一个明确的、统一编码的、可判定的语言族；
- 但等式逻辑问题的复杂度非常依赖你讨论的是**哪个片段**、**哪种签名**、**哪些限制条件**。

更适合本问题的理论提问往往是：

- 这个片段是否可判定？
- 如果可判定，已知上界是什么？
- 是否存在高效的规范形或重写系统？
- 有限反例搜索是否完备？
- 证明长度是否会爆炸？

### 4.2 一般情形下，equational theory 可以是不可判定的

这点非常关键。  
McNulty 的教材明确指出：

- “most of the equational theories are undecidable”
- 并给出了“finitely based undecidable equational theory”的构造思路。

也就是说，**不能把整个等式逻辑世界都想成一个干净的、统一可高效求解的问题**。  
在一般情形下，存在有限基但不可判定的 equational theory。

### 4.3 但受限片段又可能很好算

另一方面，不能因为一般情形复杂，就反过来说“这种问题只能靠黑箱”。  
很多受限版本其实是可处理甚至相对高效的，例如：

- 变量与项深度被严格限制；
- 只考虑特定类公理；
- 已经拥有收敛重写系统；
- 只做有限规模反例搜索；
- 只处理 ground / bounded / finite 版本。

因此，更准确的说法是：

> 这个领域既有“理论上很难甚至不可判定”的一般情形，也有“在特定片段上可以做得很好”的受限情形。

### 4.4 对 SAIR 当前题型，不应在没有专项文献核实前随便下精确复杂度结论

对于本竞赛的精确任务：

- 单前提
- 单二元运算
- 题目来自特定规模的等式库
- 公开数据分布明显经过筛选

这些因素会让“实际 benchmark 的计算难度”与“该逻辑问题的一般理论难度”不是一回事。

因此，当前最稳妥的表达是：

- 可以进行算法理论分析；
- 但**不应在没有专项文献核实前，直接声称“SAIR 这道题就是 P / NP-complete / undecidable”**。

## 5. 为什么竞赛却让我们走 Prompt 路线

### 5.1 这是竞赛接口约束，不是理论极限

从仓库里的官方公开元数据看，Stage 1 的接口本质上是：

- 使用 `evaluation` 模板；
- 把 `equation1 / equation2 / cheatsheet` 注入同一个 `complete prompt`；
- 用统一 prompt 在同一套 benchmark 上批量评测。

对应证据可见：

- `data/raw/prompt_templates.jsonl`
- `data/raw/benchmarks.jsonl`
- `docs/SAIR代数推理竞赛工程化实验计划.md`

因此我们当前做的，不是“寻找数学上最强求解器”，而是：

> 在“只能提交一个统一 complete prompt”的约束下，尽可能把可迁移的代数推理知识压缩进 prompt。

### 5.2 Prompt 路线研究的是“知识蒸馏与行为控制”

这个比赛更像是在测试三件事：

1. 你能否识别哪些结构规律对真假判断有帮助；
2. 你能否把这些规律压缩成统一文本而不导致模型崩坏；
3. 你能否在不可按题切换策略的前提下保持全局稳定性。

所以这里的 prompt 工程，本质上更接近：

- **规则压缩**
- **推理行为蒸馏**
- **统一策略设计**

而不是“替代整个自动推理领域”。

---

## 6. 为什么不能把“做 Prompt”理解成“放弃理论”

### 6.1 官方项目本身就大量依赖符号与形式化方法

ETP 官方页面与论文非常清楚地说明：

- 他们完成的是 4694 条 magma 等式之间 `22,028,942` 条 implication 的判定；
- 使用了 Lean 进行形式化验证；
- 使用了自动定理证明器、Z3、有限 magma 工具等；
- LLM 在项目里是有限度使用的，真正高效的是 ATP 等自动化工具。

这给我们一个很重要的认知纠偏：

> 数学家没有因为“无计可施”才转向 LLM；  
> 相反，在这个方向上，符号方法、自动证明、反例搜索才是主战工具。

### 6.2 LLM 更像受限接口下的“近似决策器”

放到我们的竞赛环境里，可以把 LLM 理解为：

- 一个不能外接 Lean/Z3/Prover9 的黑箱模型；
- 但它能接受一段固定文本提示；
- 我们的目标是让这段文本尽可能逼近正确的结构性判定行为。

所以当前项目更像是在研究：

> 当不能直接调用强符号工具时，怎样用 prompt 把一部分结构知识“蒸馏”进模型行为中。

---

## 7. 这也解释了为什么我们当前的工程路线是合理的

### 7.1 固定切分、baseline、错误分析，不只是工程洁癖

如果这个问题本质上是结构性决策问题，那么 prompt 研发最怕的不是“偶尔少做对一题”，而是：

- 不知道自己到底学到了什么结构；
- 不知道提升来自哪里；
- 不知道是否只是把某一批样本背下来了。

因此，我们建立的这些组件其实都和理论视角是匹配的：

- **固定切分**：防止把开发集误当测试集；
- **baseline**：给 prompt 一个最低参照系；
- **错误分析 taxonomy**：区分“规则缺失”“规则冲突”“false-filter 过弱”等不同失败机制；
- **family tagger**：把问题切成更窄的结构族；
- **offline rule assets**：把“有效但暂不宜直接写进 prompt”的规律先沉淀下来。

### 7.2 离线 family tagger 的意义，恰好在于向算法视角靠拢

我们最近从“继续堆 wording”逐步转向：

- `family tagger`
- `offline rule assets`
- `canonical axis consolidation`
- `deduplicated review set`

这一步的理论意义在于：

> 我们开始不再把所有改进都押在自然语言 wording 上，  
> 而是先把结构规律显式化、标签化、资产化。

这其实是一种**弱符号化 / 半结构化**路线，属于在竞赛约束下，向更可解释、更可复用的算法资产靠拢。

---

## 8. 对导师最适合的表述方式

如果需要向导师简洁解释，推荐使用下面这段逻辑：

1. 这个竞赛问题本质上是一个标准的等式蕴含判定问题，属于等式逻辑 / 普遍代数范畴。
2. 它当然可以做算法理论分析，但复杂度高度依赖讨论片段，不能简单概括成统一的 P/NP 问题。
3. 一般的 equational theory 甚至可能不可判定，因此理论上并不“简单”。
4. 但官方 Equational Theories Project 已经证明，这类问题非常适合形式化证明、反例构造、自动定理证明和有限模型搜索。
5. 我们当前走 prompt 路线，不是因为符号方法不存在，而是因为竞赛接口要求提交统一 `complete prompt`。
6. 因此，本项目的核心不是“盲目调 prompt”，而是在接口约束下，把结构性代数知识尽可能稳定地蒸馏进模型行为中。
7. 当前从 prompt wording 进一步过渡到 family tagger / offline assets，本质上是在把经验调参逐步转成更接近算法资产的形式。

---

## 9. 对当前项目的直接启示

### 9.1 可以继续做的事

- 保持 `P1_2_3` 作为稳态主 prompt；
- 继续扩展 `family tagger` 与离线规则资产；
- 用更细粒度结构标签替代泛化 wording；
- 把 review set 当作“局部理论假设的人工复核池”。

### 9.2 不宜做的事

- 在没有文献支撑前，声称“这个竞赛问题就是某个固定复杂度类”；
- 把 LLM 路线表述成“替代形式化方法”的方案；
- 把每个有效结构都直接写回 prompt，忽略 wording 稳定性；
- 混淆“竞赛提交约束下的最优策略”和“数学上最强算法”。

### 9.3 如果未来允许做更强的研究扩展

在不受 Stage 1 prompt 约束时，可以考虑把以下方向作为增强研究线：

- 有限 magma 自动反例搜索；
- 基于等式语法树的规范化或重写；
- implication graph 特征建模；
- 与 Lean / ATP / SMT 的离线对照验证；
- 用程序生成更可靠的“正例结构信号”，再决定是否以隐式方式注入 prompt。

## 11. 参考解读说明

本文刻意采用“保守而严谨”的说法：

- 对“这是一个标准决策问题”给出肯定；
- 对“是否能做复杂度分析”给出肯定；
- 对“是否能直接宣称它是 P/NP 某类问题”保持克制；
- 对“为什么当前仍然做 prompt”给出竞赛接口层面的解释。

这样的表述更适合导师汇报、周报、开题讨论和后续项目定位。
