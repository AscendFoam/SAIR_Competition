## 1. 文档目的

本文档用于按照实际推进顺序，回顾本项目从启动到当前阶段已经完成的工作，并尽量用非专业人士也能理解的方式解释：

- 这个项目在做什么
- 为什么要这样组织工程
- 每一阶段做了哪些事情

- 当前项目处在什么位置

需要特别说明的是，这个项目不是把竞赛当作一次性的“手工试题”，而是把它当作一个持续迭代的工程实验系统来建设。因此，除了 prompt 本身，我们还同步建设了数据链路、评测链路、错误分析、实验对比、版本管理和报告机制。

## 2. 项目背景：我们在解决什么问题

本竞赛的核心任务可以简单理解为：

给出两个代数等式 `Equation 1` 和 `Equation 2`，判断前者是否必然推出后者。

更口语化一点说，就是：

- 如果所有满足 `Equation 1` 的代数结构，也一定都满足 `Equation 2`，答案就是 `true`
- 只要能找到哪怕一个满足 `Equation 1` 但不满足 `Equation 2` 的代数结构，答案就是 `false`

这类题对大模型来说并不容易，因为它不是普通的文本匹配任务，而是同时涉及：

- 符号结构理解
- 变量与子式关系识别
- 抽象代数语义
- 多步推理
- 对“什么时候必须为真、什么时候只是不像真”的区分

因此，这个项目从一开始就不适合只靠“随便改几句 prompt”来推进，而需要一个稳定的工程框架，让每一轮尝试都可以被记录、复现和比较。

## 5. 仓库私有化与运行环境治理：为什么要做 `.gitignore`

### 5.1 为什么竞赛项目必须重视私有化

竞赛项目和普通开源项目不同。普通开源项目强调尽量公开，竞赛项目则必须谨慎区分：

- 哪些是可以公开的“工程结构”
- 哪些是不能公开的“竞赛资产”

竞赛资产通常包括：

- 官方原始数据
- 自己的 prompt 文本
- 实验报告
- 中间分析结论
- 运行日志
- API 密钥

如果这些内容直接进入 Git 仓库，很容易误传到远端平台，造成不必要的风险。

### 5.2 `.gitignore` 的治理内容

因此，专门对 `.gitignore` 做了补强，目前已屏蔽：

- `.env` 与环境变量文件
- `webpages/` 研究资料
- `data/raw/*`
- `data/interim/*`
- `data/external/*`
- `prompts/cheatsheets/*`
- `prompts/complete/*`
- `reports/experiments/*`
- `reports/release/*`
- `artifacts/candidates/*`
- `artifacts/final/*`
- `artifacts/backups/*`
- 本地配置 JSON/JSONL
- pytest cache 与临时目录

但同时保留了：

- README
- 模板文件
- example config

这样一来，仓库公开时仍然能看出结构和方法，但不会暴露真正的竞赛内容。

### 5.3 运行缓存治理

在真实使用过程中，还遇到一个非常常见但很影响体验的问题：

- Python 和 pytest 运行后会自动在仓库根目录下生成 `__pycache__`、pytest cache、临时目录等文件

这些文件虽然不影响程序功能，但会让仓库显得很乱，也更容易误提交。

后来采取了几项措施：

- 把 pytest cache 固定到 `artifacts/pytest_cache`
- 把 pytest 临时目录固定到 `artifacts/pytest_tmp`
- 在运行时设置 `PYTHONDONTWRITEBYTECODE=1`
- 在 `.gitignore` 中屏蔽常见缓存目录

这属于“工程卫生”工作。它不直接提升模型效果，但能让整个项目更稳定、更整洁，也更适合长期维护。

## 6. 公共数据接入阶段：官方数据标准化

### 6.1 为什么不能直接拿原始数据就跑

官方原始数据往往包含多个文件、多个格式和额外说明信息，直接拿来做实验会有几个问题：

- 格式不统一，不方便批量处理

## 7. 固定切分阶段：为什么要切分、为什么这样切、各集合分别做什么

这是很多非专业读者最容易疑惑的一部分，所以这里单独详细解释。

### 7.1 为什么要切分数据

如果把全部题目混在一起反复调 prompt，会出现一个严重问题：

- 你看到的结果越来越好，但你并不知道模型是真的变强了，还是只是“越来越适应你反复看到的那批题”

这在机器学习和实验设计里叫做“评测污染”或“过拟合风险”。

简单来说，如果同一批题既被拿来想规则，又被拿来打分，那么分数会越来越“不可信”。

因此，必须把数据切成不同用途的子集，让不同阶段做不同事情。

### 7.2 为什么要做“固定切分”

不仅要切分，而且要“固定切分”。

固定切分的意思是：

- 一旦确定了 `smoke/dev/holdout/audit` 的成员，就不要每次实验都随机重抽

原因是如果每轮都换一批题：

- 今天得分高，不知道是不是因为 prompt 真变好了
- 也可能只是今天抽到的题刚好更容易

固定切分的价值在于：

- 每次都是在同一组题上比较
- 差异可以更可信地归因到 prompt 或规则变化
- 报告里的对比更有意义

### 7.3 为什么切成 `smoke/dev/holdout/audit`

这里不是随便取了几个名字，而是对应不同工作阶段的不同用途。

#### `smoke`

作用：

- 最小规模、最快速的冒烟测试集

为什么需要它：

- 每次改 prompt 都跑全量会很慢、很贵
- 很多坏 prompt 一眼就能在小集合上看出来

它适合回答的问题是：

- 这版 prompt 是否能正常输出
- 是否出现了明显塌缩，例如几乎全 `false` 或几乎全 `true`
- 是否值得继续进入更大规模测试

当前 `smoke` 规模：

- 共 `64` 题
- `true=31`
- `false=33`
- 来源分布包含 `normal/hard1/hard2`

这意味着它是一个小而均衡的“体检集”。

#### `dev`

作用：

- 主要开发集

为什么需要它：

- `smoke` 太小，容易偶然波动
- 需要更大规模数据来观察 prompt 的稳定趋势

它适合回答的问题是：

- 这类规则在更大样本上是否还成立
- 某类错误是否真的在下降
- 某种改法是不是只在小样本上看起来有效

#### `holdout`

作用：

- 保留评估集

为什么需要它：

- 如果你经常盯着一批数据改 prompt，那批数据就不再是“客观评委”

`holdout` 的意义就是：

- 平时尽量少用
- 关键节点再用来检查“当前方法是否真的值得信”

它更像是项目中的“阶段考试”，不是每天练习题。

#### `audit`

作用：

- 额外人工抽查和分析集

为什么需要它：

- 有些时候我们不是只想知道“分数”，还想知道“错在哪”
- 需要一批适合做人工审阅、规则核查、边界案例复盘的样本

它更像“质检样本集”。

### 7.4 为什么是现在这个切分比例

当前切分如下：

- `smoke = 64`
- `dev = 824`
- `holdout = 254`
- `audit = 127`

这种安排背后的考虑是：

1. `smoke` 要足够小，才能频繁跑。  
如果 smoke 太大，每次迭代成本会过高。

2. `dev` 要足够大，才能承担主要实验工作。  
因为大部分迭代都要在这里看趋势。

3. `holdout` 要保留一定体量，才能在关键时刻做相对可信的验证。  
如果太小，就没有代表性。

4. `audit` 不需要特别大，但要足够做人工检查。  
它重在“可读”和“可分析”，不重在压分。

### 7.5 为什么切分后还要看标签分布和来源分布

切分不是把题目随机分出去就结束了，还要检查每个集合内部是否大致均衡。

如果一个集合里：

- `true` 太多
- 或 `false` 太多
- 或全是 `normal`
- 几乎没有 `hard`

那么这个集合就容易给出误导性结论。

当前 `smoke`、`dev`、`holdout`、`audit` 都保留了：

- `true/false` 的相对平衡
- `normal/hard1/hard2` 的来源覆盖

这保证了它们更像“缩小版总体”，而不是偏科样本。

## 8. Baseline 阶段：为什么先做 baseline，baseline 是怎么设计的

### 8.1 什么是 baseline

可以把 baseline 理解成“最基础参照物”。

在竞赛或模型实验中，如果没有 baseline，就会有两个问题：

1. 你不知道当前结果到底算好还是差。
2. 你不知道新方法到底是真的有效，还是只是比自己之前随手写的版本好一点。

所以 baseline 的作用不是追求最强，而是提供参照系。

### 8.2 为什么这个项目必须先做 baseline

这个项目的 prompt 设计很容易出现“感觉上更聪明，但实际没更好”的情况。  
如果没有 baseline，就会经常出现：

- 看起来 reasoning 更长了
- prompt 更复杂了
- 但结果并没有比最简单的规则强

baseline 就像量尺，帮助我们判断：

- 一个复杂方案是否真的值这个复杂度
- 某个 prompt 是不是其实退回了很低水平

### 8.3 目前已经建立的 baseline

项目中已经实现了四个 baseline：

- `always_false`
- `always_true`
- `canonical_match`
- `structural_v1`

它们分别代表不同层级的“最低参照”。

#### `always_false`

意思很简单：

- 不管题目是什么，一律输出 `false`

这个 baseline 看似很笨，但它很有用。  
因为如果数据里 `false` 稍微多一些，那么“永远输出 false”就可能拿到不算太差的总准确率。

所以它能告诉我们：

- 一个模型如果连 `always_false` 都赢不了，就说明它几乎没有真正学到东西

#### `always_true`

对应地：

- 不管题目是什么，一律输出 `true`

它的作用是对称参考，帮助我们看：

- 数据分布更偏哪边
- 模型是否出现了“全真塌缩”

#### `canonical_match`

这个 baseline 的意思是：

- 只有当两个等式在变量重命名后完全等价时，才输出 `true`

例如：

- `x = y * z`
- `a = b * c`

这类题在结构上是同一个模板，只是变量名字不同。

这个 baseline 的作用是提供一个“纯形式匹配”的下限：

- 它不会做真正推理
- 只会识别“其实就是同一句话”

#### `structural_v1`

这是目前 baseline 里最有实际价值的一个。  
它不是大模型，也不懂真正的代数语义，但会使用一些轻量结构规则，例如：

- 比较两个等式的变量数量
- 比较运算符数量
- 比较左右侧结构形状
- 判断左右两边是否存在明显的“更强 / 更复杂”关系

它的意义在于：

- 它比“全 true / 全 false”更接近真实推理
- 它为后续 prompt 设计提供了结构启发
- 它能提醒我们：有些提升其实只是简单结构规则就能解释

### 8.4 baseline 是在哪个集合上来的

当前 baseline 的代表性结果主要是在 `holdout` 上统计的。

原因是：

- baseline 是项目的基础参照物
- 需要在一个相对独立、不过度参与日常调参的集合上给出结果

在 `holdout` 上，`structural_v1` 的表现为：

- Accuracy：`0.5787`
- True Accuracy：`0.4400`
- False Accuracy：`0.7132`

这个结果说明：

- 简单结构规则已经比“全真 / 全假”强不少
- 任务并不是完全只能靠大模型“玄学发挥”
- prompt 如果想证明自己有价值，至少要知道自己和这种结构启发式相比提升了什么

### 8.5 baseline 的真正作用

这套 baseline 后来承担了三种作用：

1. 作为“最低参照”。  
帮助判断一版 prompt 是否连最简单规则都不如。

2. 作为“结构启发来源”。  
帮助发现哪些结构特征值得写进 prompt 或错误分析。

3. 作为“实验 sanity check”。  
如果某个复杂 prompt 的表现和 `always_false` 很像，那往往说明它已经塌缩了。

## 9. 错误分析体系：为什么不能只看准确率

### 9.1 为什么只看总分会误导

如果只看一个数字，例如 Accuracy，很容易被误导。

举个简单例子：

- 一个 prompt 总准确率可能和另一个一样
- 但前者是“几乎全 false”
- 后者是“true 和 false 都还能判断一些”

这两者表面总分相同，但价值完全不同。

因此，项目里从比较早的时候就建立了错误分析体系，目的不是只看“错了多少”，而是看“错的类型是什么”。

### 9.2 当前错误分析的核心类别

当前错误分类体系包括：

- `FORMAT`
- `RULE_MISSING`
- `RULE_CONFLICT`
- `FALSE_FILTER_WEAK`
- `HARD_COMPOSITION`
- `PROMPT_AMBIGUOUS`
- `OVERCOMPRESSION`
- `MODEL_SPECIFIC`

其中在实际 prompt 迭代中最常用的是前五类。

### 9.3 外行可理解的错误类型解释

#### `FORMAT`

意思是：

- 模型输出格式不对，系统无法稳定解析成 `true` 或 `false`

这不是“推理错了”，而是“答题格式不合规”。

#### `RULE_MISSING`

意思是：

- 模型本来应该判 `true`
- 但 prompt 里缺少某条关键规则，导致它没能触发正确判断

可以把它理解成：

- 不是模型完全乱来
- 而是我们的提示规则没有覆盖到这类题

#### `RULE_CONFLICT`

意思是：

- prompt 里不同规则之间互相打架了
- 或者某条规则覆盖过头，误伤了别的情况

这类错误通常意味着：

- prompt 不是“规则太少”
- 而是“规则之间关系不对”

#### `FALSE_FILTER_WEAK`

意思是：

- prompt 太容易输出 `true`
- 没有足够谨慎地拦住一些其实不成立的题

直观理解就是：

- “放水太多”

#### `HARD_COMPOSITION`

意思是：

- 这道题不是靠一条单独规则就能搞定
- 需要多条规则组合，或者需要多步推理

这类错误通常出现在更难的题上，说明：

- prompt 不是完全没方向
- 而是模型没完成“把几步拼起来”的过程

### 9.4 错误分析体系的价值

这套体系后来成为 prompt 迭代的方向盘。

因为它能帮助我们回答：

- 这版 prompt 是规则不够，还是规则冲突
- 是过度保守，还是过度乐观
- 是结构理解问题，还是多步组合问题

如果没有这些分类，实验就只能停留在：

- “这版分数高一点”
- “那版分数低一点”

而无法知道下一步该改哪里。

## 10. Prompt 工程基础阶段：官方模板恢复与完整 prompt 机制

### 10.1 为什么要恢复官方模板

竞赛里 prompt 不是随便写的文本，而是实验对象本身。  
如果连官方模板都没有恢复，那么后续就很难判断：

- 我们现在的实验和官方设置差了多少
- 哪些变化是基于官方模板进行的
- 哪些结果可以更好地对照官方公开信息

因此，根据 `data/raw/prompt_templates.jsonl` 恢复了：

- `prompts/templates/evaluation.jinja2`

这让仓库中有了一个可以追溯到官方记录的 prompt 模板。

### 10.2 为什么又建立了严格模板

官方模板更像“原始还原版”，但实际工程中还需要一个更适合自动化评测的版本，于是建立了：

- `prompts/templates/evaluation_strict_v1.jinja2`

它要求：

- 第一行只输出 `true` 或 `false`
- 不要额外标签
- 不要先写解释

原因很简单：

- 对人来说，多写解释可能没问题
- 对批量评测系统来说，格式越严格越稳定

### 10.3 为什么要分成 template / cheatsheet / complete prompt 三层

这是项目中非常关键的工程设计。

#### template

作用：

- 提供固定骨架，例如任务定义、输出格式、变量占位符

#### cheatsheet

作用：

- 放具体规则、提醒、推理约束

#### complete prompt

作用：

- 把 template 和 cheatsheet 合成后，得到可以直接送给模型的最终 prompt

这样设计的好处是：

1. 可以单独改规则，而不必每次重写整个 prompt。
2. 可以共享同一个模板，比较不同规则内容。
3. 更容易记录“这版 prompt 到底改了什么”。

这就像搭积木，而不是每次从头捏一个模型。

### 10.4 新增官方 cheatsheet 样例后，对“官方资产”的理解发生了什么变化

在最初阶段，我们拿到的、最容易直接利用的官方 prompt 相关材料，是
`data/raw/prompt_templates.jsonl` 中记录的模板信息。  
因此当时项目的重点是先把官方 `evaluation` 模板恢复出来，再建立适合本地自动化评测的 strict 版本。

后来仓库中又加入了几份官方提供的 cheatsheet 样例：

- `data/raw/cheatsheet_balanced.md`
- `data/raw/cheatsheet_counterexample_first.md`
- `data/raw/cheatsheet_example_fast_filters.md`

这几份文件带来的一个重要认知更新是：

- 它们并不只是“几条简短规则的摘要”
- 它们通常同时包含任务 framing、占位符、推理步骤、输出格式要求
- 从工程上看，它们更接近“官方 prompt 原型”或“官方 prompt 风格样例”

这意味着我们对“官方资产”的理解，从原来的“只有官方 template 元数据”进一步扩展为：

- 一类资产是官方 template 元数据
- 另一类资产是官方 cheatsheet / prompt 风格样例

但这个变化并不意味着当前主线策略要被推翻。原因是这些官方样例往往：

- 更偏向完整解释型输出
- 容易要求模型生成 `REASONING`、`PROOF`、`COUNTEREXAMPLE` 等长文本
- 与当前 DeepSeek 本地 strict 评测链路中的“首行稳定输出 `true/false`”目标并不完全一致

因此，到目前为止，这些官方样例的正确定位更适合是：

- 用来校准我们对官方 prompt 风格的理解
- 用来设计新的 prompt 家族实验分支
- 用来作为后续“官方风格 strict 改写版”的参考来源

而不是直接替换当前已经验证过稳定性的本地主线 prompt。

## 11. 本地评测链路阶段：DeepSeek 接入、解析器与运行问题修复

### 11.1 为什么要做本地 runner

如果每次测试 prompt 都靠手工复制粘贴到网页聊天框里，会有很多问题：

- 速度慢
- 不可复现
- 不方便批量统计
- 不方便保存原始输出
- 不方便做错误分析

因此，项目中实现了本地 runner：

- `src/sair_competition/eval/openai_compatible.py`
- `src/sair_competition/eval/local_runner.py`
- `src/sair_competition/config/env.py`

它允许我们通过 `.env` 中的 DeepSeek 配置，批量调用模型完成 smoke 评测。

### 11.2 为什么要特别做输出解析器

大模型即使知道答案，也不一定按你想要的格式输出。  
它可能输出：

- `true`
- `TRUE`
- `VERDICT: TRUE`
- `The answer is false`

对于人来说，这些都能理解；但对自动脚本来说，必须明确怎么解析。

因此实现了输出解析器及测试：

- `src/sair_competition/eval/parser.py`
- `tests/test_eval_parser.py`
- `tests/test_eval_parser_more_formats.py`
- `tests/test_eval_parser_verdict.py`

这一步的作用是：

- 尽量降低格式波动带来的误判
- 让批量评测更稳定

### 11.3 运行环境问题修复

实际使用中遇到过：

- `No module named 'sair_competition'`

这类问题。后来明确了项目的标准运行方式：

- 安装 editable package
- 或在运行前设置 `PYTHONPATH=src`

并统一使用：

- `conda` 的 `DLEnv`

这一步看似技术细节，但非常关键，因为它解决的是“这个项目到底怎么稳定跑起来”的问题。

## 12. 第一轮 smoke 验证：打通实验闭环

### 12.1 为什么先做 smoke，而不是一上来跑全量

这和切分逻辑类似。  
一个 prompt 刚写出来时，最常见的问题不是“差 1% 分数”，而是：

- 输出格式不对
- 整体塌缩成全 `false`
- 某条新规则把行为推歪了

这类问题没必要用大规模数据才能发现。  
所以先用 `smoke` 做快速体检，成本最低，反馈最快。

### 12.2 最早的链路验证结果

使用示例 complete prompt 对 `smoke` 小样本做了首轮验证。  
在 `limit=8` 的试跑中得到：

- Accuracy：`0.75`
- True Accuracy：`0.0`
- False Accuracy：`1.0`

这个结果并不表示模型很强，恰恰相反，它说明：

- 链路跑通了
- 但模型明显倾向于判 `false`

这正是 smoke 的价值：

- 不是要给出最终分数
- 而是尽快发现“当前行为长什么样”

### 12.3 第一批 prompt smoke 候选

随后陆续建立了早期候选：

- `P0_smoke_probe`
- `P1_1_0_strict_empty_probe`
- `P1_1_1_strict_first_draft_smoke`
- `P1_2_0_smoke_iter1`
- `P1_3_0_smoke_iter2`

其中最重要的阶段结论来自：

- `reports/experiments/smoke_prompt_compare/comparison.md`

### 12.4 第一阶段最重要的结果：`P1_2_0`

在早期实验中：

- `P1_1_1` 基本接近全 `false`
- `P1_3_0` 也退回全 `false`
- 只有 `P1_2_0_smoke_iter1` 真正打破了这种塌缩

`P1_2_0_smoke_iter1` 指标：

- Accuracy：`0.5156`
- True Accuracy：`0.6129`
- False Accuracy：`0.4242`
- Balanced Accuracy：`0.5186`

这一步的意义不在于它已经最好，而在于它第一次证明：

- 这个任务上，prompt 是可以被调出“非全 false”行为的
- 项目具备了继续优化的现实基础

## 13. 第二阶段：显式 false-filter 路线失败

### 13.1 当时为什么会想到做 false-filter

在 `P1_2_0` 后，分析发现一类明显问题：

- 有些题本来不应该判 `true`
- 但 prompt 在 shared-variable 情况下容易“过度相信结构相似性”

于是自然想到：

- 那就加一层更明确的“false-filter”来拦截这些假阳性

### 13.2 实验了哪些分支

这一阶段主要尝试了：

- `P1_2_1_family_d_light_false_filter`
- `P1_4_0_family_d_false_filter`
- `P1_4_1_family_d_false_filter_softened`
- `P1_4_2_family_d_false_filter_score`

对应报告：

- `reports/experiments/family_d_false_filter_compare/comparison.md`

### 13.3 为什么这条路被判定为失败

结果非常一致：

- 这些分支几乎都退化成全 `false`
- Accuracy 看上去没掉很多
- 但 True Accuracy 直接掉到 `0.0`

这说明：

- 它们并不是“更聪明地判断 false”
- 而是“把 true 也一起杀掉了”

这一步得出的模型行为结论非常重要：

- DeepSeek 对显式 false-filter 很敏感
- 一旦把“保守判 false”写得过重，就容易整体塌缩

这直接改变了后面的主线策略。

## 14. 第三阶段：implicit guardrail 路线成功建立稳态主分支

### 14.1 策略如何调整

既然显式 false-filter 会让模型塌缩，那么新的思路就是：

- 不把 false-filter 做成主决策层
- 只把它写成短、弱、隐式的 guardrail
- 保持 prompt 主体仍由正向规则和输出纪律构成

### 14.2 这一阶段的主要候选

- `P1_2_2_implicit_guardrail_v1`
- `P1_2_3_implicit_guardrail_v2`

对应报告：

- `reports/experiments/implicit_guardrail_compare/comparison.md`

### 14.3 为什么 `P1_2_3` 成为当前稳态主分支

`P1_2_3_implicit_guardrail_v2` 的结果为：

- Accuracy：`0.5469`
- True Accuracy：`0.2258`
- False Accuracy：`0.8485`
- Balanced Accuracy：`0.5371`

相较于 `P1_2_0`：

- Accuracy 增加 `+0.0312`
- Balanced Accuracy 增加 `+0.0186`

更重要的是，它没有像显式 false-filter 那样完全塌缩。  
这说明：

- “短、弱、隐式 guardrail”是 DeepSeek 更能接受的风格

因此 `P1_2_3` 被保留为当前稳态主分支。

## 15. 第四阶段：围绕 `P1_2_3` 的高召回挖掘实验

### 15.1 为什么还要继续挖

虽然 `P1_2_3` 稳定，但它还有明显短板：

- 很多该判 `true` 的题仍被漏掉
- 特别是 `RULE_MISSING` 和 `HARD_COMPOSITION`

所以项目进入下一阶段：

- 不轻易动稳态主分支
- 但可以开更激进的高召回支线来“挖规则”

### 15.2 这一阶段的主要实验

- `P1_2_4_rule_missing_hard_composition`
- `P1_2_5_minimal_rule_missing_hard_composition`
- `P1_2_6_collapse_priority`
- `P1_2_7_light_anti_bias`
- `P1_2_8_narrow_singleton_families`

相关报告：

- `reports/experiments/P1_2_3_to_P1_2_7_iter_compare/comparison.md`
- `reports/experiments/P1_2_3_P1_2_5_P1_2_8_compare/comparison.md`

### 15.3 这些分支分别说明了什么

#### `P1_2_4`

对 rule-missing / hard-composition 的表达更直接，但再次把模型推向近似全 `false`。  
说明 DeepSeek 对 wording 非常敏感。

#### `P1_2_5`

这是这轮中最有价值的高召回支线：

- Accuracy：`0.5781`
- True Accuracy：`0.6452`
- False Accuracy：`0.5152`
- Balanced Accuracy：`0.5802`

它的意义是：

- 证明确实可以补回很多漏判的 `true`

但它的问题也很明显：

- 把不少不该通过的题也放过去了

所以它不适合作为主分支，更适合当“规则发现器”。

#### `P1_2_6`

把 collapse 优先级写得太强，模型又摆向近乎全 `true`。

这说明：

- 不只是“过强 false-filter”危险
- “过强 true-trigger”同样危险

#### `P1_2_7`

哪怕只是更轻微的 anti-bias wording，DeepSeek 也可能退回近乎全 `false`。

这进一步证明：

- `P1_2_3` 附近的 wording 稳定区间很窄

#### `P1_2_8`

把从 `P1_2_5` 中挖出的窄 singleton 结构族直接写回 prompt。  
逻辑上它们是很干净的，但模型行为仍然被推回近似全 `false`。

这说明：

- “规则逻辑上安全”不等于“写进 prompt 后模型还能稳定使用”

## 16. 规则挖掘阶段：从 `P1_2_5` 抽取真正有价值的结构族

### 16.1 为什么要把 `P1_2_5` 当成“挖掘器”

虽然 `P1_2_5` 不够稳，但它成功翻正了一批题。  
这就意味着它并非没有价值，而是：

- 它包含了一些有用信息
- 只是这些信息目前还没有被“安全地”写成主 prompt

所以后续策略不是直接继承 `P1_2_5`，而是把它当成：

- 高召回挖掘分支

也就是用它来回答：

- 它究竟补对了哪些题
- 这些题有没有共通结构
- 这些结构是否能在更大数据上验证为安全

### 16.2 已挖出的高置信结构族

在 `public_all` 上验证后，确认了两个很干净的 singleton 子族。

#### `singleton_left_same_lhs`

含义：

- `Equation 1` 是 `x = T`
- `x` 不出现在 `T` 里
- `Equation 2` 仍然以 `x` 为左边

统计结果：

- `161 true / 0 false`

这说明只要碰到这种结构，几乎可以确定它属于“安全的真例家族”。

#### `singleton_left_binary_lhs_with_x`

含义：

- `Equation 1` 是 `x = T`
- `x` 不出现在 `T` 里
- `Equation 2` 左边虽然不是单变量，但仍然包含 `x`

统计结果：

- `82 true / 0 false`

这类结构也非常干净。

### 16.3 为什么这些规则没有直接写回主 prompt

这是一个很关键、也很反直觉的结论。

按逻辑看，这些规则非常安全；但在 DeepSeek 上，一旦把它们写得更显式，模型整体行为却可能被推回近似全 `false`。

所以目前项目采取的策略是：

- 把它们沉淀成“离线规则资产”
- 记录在本地规则注册表中
- 未来更可能通过 tagger、feature 或更隐式的利用方式使用
- 而不是直接改坏当前稳态主 prompt

这也是工程化方法的一个体现：

- 我们不只看“规则本身对不对”
- 还看“模型能不能稳定地按我们想要的方式使用它”

## 20. 上一次记录之后新增的关键工作：把“离线结构资产”真正做成闭环

上一版文档停留的位置，主要还是“方向已经选定”：已经知道要保留 `P1_2_3` 作为稳态主线，也知道要把重心从大规模改 wording 转向离线结构标签与规则资产。  
而从那之后新增的一大批工作，核心不是再开很多 prompt 分支，而是把这条新路线真正工程化落地，让它可以稳定复用、可统计、可审查、可继续扩展。

更直白一点说，上一阶段解决的是“该往哪走”；这一阶段解决的是“怎样把这条路修成真的能反复走的路”。

### 20.1 为什么不能停留在“打了 family tag 就够了”

仅仅给样本打上结构标签，还不足以支撑后续迭代。原因有三个：

1. 很多标签之间高度重叠。  
   例如同一个样本，可能同时属于 `singleton collapse`、`disjoint sides` 和 `constant-operation candidate`。如果直接看标签计数，会把同一批样本重复记账，容易造成“发现了很多机会”的错觉。

2. 标签本身不回答“值不值得推进”。  
   一个标签即便是结构上成立，也不代表它就适合拿去做 prompt 注入。我们还需要知道：
   - 它命中了多少题
   - 命中题里 `true/false` 怎么分布
   - 当前主线模型在这批题上是否真的经常出错
   - 它是应该继续观察，还是已经值得进入下一步程序化利用

3. 宽标签常常把真例和假例混在一起。  
   如果一个大标签里面同时混有稳定真例和稳定假例，那么它就不适合作为直接可用规则。必须先继续拆细，直到同一桶里的样本在语义上更一致。

因此，这一阶段新增工作的目标，就是把“标签”升级为“资产”，再把“资产”整理成“可复核的主轴”，最后把需要人工判断的部分收敛成小而明确的 review set。

### 20.2 离线规则资产链路落地：从标签统计走向“可管理的规则资产”

为了解决上面的问题，项目把离线规则资产链路补齐了。核心做法是：

1. 先在带标签的数据上，结合当前主线 `P1_2_3` 的错误分析结果，构建 `offline rule assets`。  
   每条资产不再只是一个标签名，而是同时携带：
   - 对应的结构触发条件
   - 支撑样本数量
   - 其中 `true/false` 的分布
   - 当前主线在该切片上的准确率
   - 是否值得继续推进的机会分数
   - 当前建议动作，例如继续观察、扩 family tagger，还是准备程序化正信号

2. 再把这些资产附着回现有预测结果中，审计它们在主线漏判中的表现。  
   这样做的意义是：不再只问“这个结构是否存在”，而是进一步问“这个结构是否正好对应了主线当前的漏判区域”。

3. 最后输出机器可读的资产文件和人可读的报告，作为后续 review 和扩展的统一入口。

这一步的本质，是把过去比较“经验主义”的判断，转成一套可度量、可排序、可回溯的资产管理系统。

### 20.3 为什么还要做 canonical axes：解决“同一批样本被多个名字重复表示”的问题

资产建立之后，又出现了第二个工程问题：  
同一批样本可能同时命中多条资产。

例如，一些非常典型的真例，既满足：

- `OA_TRUE_SINGLETON_LHS_TO_BINARY`
- 也满足 `OA_TRUE_DISJOINT_VAR_BINARY`
- 还满足 `OA_TRUE_CONSTANT_VAR_BINARY`

如果直接把这三条都当成三个独立“战果”，就会出现严重的重复记账问题。  
这不仅会误导统计，也会让后面的人工复核工作量被人为放大。

因此，项目新增了 `canonical axes` 这一步。  
它的作用可以理解为：

- 先把“完全命中同一批样本”的资产合并成同一条主轴
- 再恢复这些主轴之间的父子层级关系
- 最后保证每条 review row 只被分配给“最具体、最窄、最值得人工看的那一条轴”

这一步非常像数据分析中的“去重归并”。  
它解决的不是模型性能问题，而是分析口径问题。  
没有这一步，后面即使人工复核做得很认真，也会因为样本重复出现在多个桶里而失去清晰性。

### 20.4 从“很多资产”到“少量主轴”：为什么这能让人工复核变得可做

在这一阶段中，项目先后经历了两种视图：

- 较早的版本中，离线资产被整理成 `9` 条资产、`4` 条 canonical axes、`13` 条去重 review rows
- 到当前最新的 `day2_v2` 版本，随着某些宽轴继续细切，已经发展为 `13` 条资产、`7` 条 canonical axes、`13` 条去重 review rows

这里看起来“资产数更多了”，但这并不表示系统变乱了。  
相反，这恰恰说明：

- 之前是把一些本来不同的结构子型暂时压在同一个桶里
- 现在是把它们拆开，让每条轴的含义更清楚

而 review row 数量仍然维持在 `13` 条，没有随资产数量一起膨胀，这就说明去重机制是有效的：  
我们不是制造了更多待审工作，而是在不增加人工负担的前提下，把同一批重点样本描述得更细、更准。

### 20.5 结构化人工复核体系落地：为什么必须有 review template 和 checklist

离线资产和 canonical axes 产生之后，项目又新增了一套结构化人工复核流程，核心产物包括：

- `annotation_checklist.md`
- `p0_p1_review_template.md`
- 若干面向具体轴的复核 prompt 与 verdict 文档

之所以要做这一步，是因为有些判断不能只靠统计数字给出。  
比如：

- 某条轴里的样本是否真的属于同一个结构机制
- 某个边界特征是“有用的细分信号”，还是“只是碰巧一起出现的噪声”
- 某条规律更适合继续扩展 family tagger，还是已经足够稳定，可以准备后续更隐式的程序化利用

这类问题需要看具体样本，做“结构层面”的人工判断。  
为了避免每次复核都变成随意讨论，项目把复核问题模板化、清单化，让每轮复核都有同一套问题框架：

- 当前 axis 是否正确
- 是否应该拆 child axis
- 哪些 boundary tags 值得保留
- 下一步动作是继续观察、扩 tagger，还是准备程序化正信号

这一步的价值在于把“人工经验”也纳入工程流程，而不是停留在聊天记录或个人印象里。

### 20.6 `TARGET_SHARED_LHS_AND_NEW_VARS` 的细切：把混合真假的大桶拆成更窄子族

上一次记录之后，一个非常重要的推进，是对 `TARGET_SHARED_LHS_AND_NEW_VARS` 这条宽标签做了更细的拆分。

之所以要拆，是因为这条大标签虽然都共享“同左边 + 引入新变量”这一表面结构，但内部并不统一：

- 有些是真正稳定的真例子型
- 有些则恰好是稳定的假例边界子型

如果继续把它们混在一起，就会出现非常危险的情况：  
你会以为“这个家族整体值得推动”，但实际上它同时包含了应该保留的正向规律和必须警惕的反例边界。

因此，项目把它继续切成了更窄的子型，例如：

- `TARGET_SHARED_LHS_AND_NEW_VARS_SINGLETON_SOURCE`
- `TARGET_SHARED_LHS_AND_NEW_VARS_FULL_CARRY_SINGLE_X`
- `TARGET_SHARED_LHS_AND_NEW_VARS_X_REPEAT_SINGLE_ANCHOR`
- `TARGET_SHARED_LHS_AND_NEW_VARS_X_HEAVY_TWO_ANCHOR`

在当前 `smoke` 固定切分上的结果是：

- `SINGLETON_SOURCE`：`2` 题，全部为 `true`
- `FULL_CARRY_SINGLE_X`：`1` 题，为 `true`
- `X_REPEAT_SINGLE_ANCHOR`：`2` 题，全部为 `false`
- `X_HEAVY_TWO_ANCHOR`：`2` 题，全部为 `false`

这一步的意义非常大，因为它第一次把一条“宽标签中的真假混合问题”拆成了方向清晰的子桶：

- 哪些是可以继续保留的真例子型
- 哪些是应该显式视为边界假例的结构

这也直接证明了项目当前路线的合理性：  
不是所有规律都该写回 prompt，而是应该先拆到足够窄、足够纯，才谈后续利用。

### 20.7 `OA_TRUE_TARGET_LHS_AMPLIFICATION` 的细切：从“看上去都像真例”到“只推进最稳的那一半”

本阶段最典型、也最能体现方法论升级的一项工作，是对 `OA_TRUE_TARGET_LHS_AMPLIFICATION` 的再次细切。

最初，这条轴包含 `3` 条 `smoke` 中的真例：

- `normal_0040`
- `normal_0489`
- `normal_0920`

它们的共同点是：

- `Equation 1` 左侧是一个会发生塌缩的单变量
- `Equation 2` 右侧对这个左侧变量做了明显“放大”
- 同时还保留了来自 `Equation 1` 的某些非左侧变量锚点

从主线上看，这 `3` 条题都被 `P1_2_3` 漏判为 `false`，因此它们很像是一个值得推进的真例家族。  
但项目没有直接把它当成“整体可注入规则”，而是又做了一轮更细的人工复核。

复核的核心问题是：  
这 `3` 条题虽然都像“LHS amplification”，但它们是否真的属于同一强度、同一可靠性的结构子型？

最终结论是：不能混在一起推进。

于是，项目把它拆成了两个更窄的子型：

- `TARGET_LHS_AMPLIFICATION_MULTI_ANCHOR`
- `TARGET_LHS_AMPLIFICATION_SINGLE_ANCHOR`

对应关系如下：

- `MULTI_ANCHOR`：
  - `normal_0040`
  - `normal_0489`
- `SINGLE_ANCHOR`：
  - `normal_0920`

之所以这样拆，是因为前两条样本都保留了两个及以上的非左侧锚点，结构支撑更强；  
而 `normal_0920` 虽然也是 `true`，但只保留了一个锚点，证据强度明显更弱，不能和前两条同强度对待。

### 20.8 这次 amplification 细切为什么重要

这个细切之所以重要，不只是因为多了两个标签，而是因为它很好地体现了项目当前的工程原则：

1. **保留父轴，不轻易丢失上层机制视角。**  
   `OA_TRUE_TARGET_LHS_AMPLIFICATION` 没有被删掉，也没有简单 merge 回父轴，而是作为上层 umbrella 继续存在。

2. **对子型区别对待，而不是“一刀切地一并推进”。**  
   `MULTI_ANCHOR` 被视为更高置信的可推进子型；`SINGLE_ANCHOR` 暂时只保留为边界观察子型。

3. **先走离线和程序化路线，不急着改主 prompt wording。**  
   这次细切没有去碰 `P1_2_3` 主 prompt 的 wording，而是优先把变化落在：
   - `family_tagger.py`
   - `offline_rule_assets.py`
   - 测试文件
   - 离线资产、主轴、review set 产物

换句话说，这一步不是“再写一句 prompt”，而是“先把规则理解得更准，再决定将来怎么用”。

### 20.9 当前最新版 `day2_v2` 链路的最新结果

在这次 amplification 子型细切完成后，项目重跑了整条 `day2_v2` 离线链路，当前得到的关键结果是：

1. 在 `smoke_tagged_day2_v2` 上：
   - `TARGET_LHS_AMPLIFICATION`：`3` 条，全部 `true`
   - `TARGET_LHS_AMPLIFICATION_MULTI_ANCHOR`：`2` 条，全部 `true`

   - 子资产 `OA_TRUE_TARGET_LHS_AMPLIFICATION_MULTI_ANCHOR`：`support_true=2`
   - 子资产 `OA_TRUE_TARGET_LHS_AMPLIFICATION_SINGLE_ANCHOR`：`support_true=1`

4. 在 `canonical axes` 中：

     - `OA_TRUE_TARGET_LHS_AMPLIFICATION_MULTI_ANCHOR`
     - `OA_TRUE_TARGET_LHS_AMPLIFICATION_SINGLE_ANCHOR`

5. 在最新 review set 中：
   - `MULTI_ANCHOR` 对应 `2` 条 `P0` review rows，建议动作是 `prepare_programmatic_positive_signal`
   - `SINGLE_ANCHOR` 对应 `1` 条 `P0` review row，建议动作是 `expand_family_tagger`

这意味着项目已经不再只是笼统地说“amplification 这类题值得关注”，而是已经能很具体地说：

- 哪一类 amplification 可以更积极推进
- 哪一类 amplification 还只能保守观察

## 22. 根据《下一周执行清单》更新后的接下来工作

`docs/下一周执行清单.md` 写成时，项目还处在“刚建立 review 机制、准备扩 tagger”的阶段。  
而现在回头看，其中相当一部分工作已经完成或部分完成，包括：

- review set 与 checklist 体系已经建起来了
- 一轮高优先级复核已经推动出了新的结构拆分
- `family_tagger` 已经吸收了 `TARGET_SHARED_LHS_AND_NEW_VARS` 和 `TARGET_LHS_AMPLIFICATION` 的更细子型
- `offline assets -> canonical axes -> review set` 已经按新标签重建到 `day2_v2`

因此，接下来的工作不应该再机械照搬原清单，而应当在原清单基础上做“阶段更新”。  
当前最合理的下一步，建议按下面顺序推进。

### 22.1 第一优先级：把最新子型拿到更大固定切分上验证泛化性

这是现在最关键的一步。

原因是：  
目前 `MULTI_ANCHOR / SINGLE_ANCHOR` 以及若干 `TARGET_SHARED_LHS_AND_NEW_VARS` 子型，都是在 `smoke` 上整理出来的。  
`smoke` 的优点是快，但它终究只有 `64` 题。  
如果一个结构规律只在 `smoke` 上成立，而到更大的 `dev` 或 `holdout` 就不稳定，那么它就不能被视为真正可靠资产。

因此，接下来最应该做的是：

- 先在 `dev` 固定切分上重新跑最新 `family_tagger`
- 统计这些新子型在更大样本上的：
  - 样本数
  - `true/false` 分布
  - 是否仍然与主线漏判强相关

这里优先推荐先做 `dev`，因为：

- 它比 `smoke` 大得多，更能看泛化性
- 又不像 `holdout` 那样珍贵，不适合频繁消耗

### 22.2 第二优先级：继续完成剩余高价值主轴的人工复核

虽然 amplification 这条轴已经做了很关键的细切，但当前 review frontier 里仍有几条高价值主轴还需要继续推进：

- `OA_TRUE_TARGET_SHARED_NEW_VARS_SINGLETON_SOURCE`
- `OA_TRUE_SINGLETON_WITH_TARGET_SHARED_LHS`
- `OA_TRUE_DISJOINT_BINARY_BINARY`

原因是：

- 它们都已经被当前离线链路识别为高优先级 review bucket
- 但还没有像 amplification 那样完成足够明确的“哪些继续推进、哪些继续观察”的子型分层

如果不继续做这一步，项目就会出现一种不平衡：

- amplification 这条线已经很细
- 其他同等级重点轴却还停在较宽的描述层面

这样后续做程序化利用时，决策标准会不一致。

### 22.3 第三优先级：围绕 `MULTI_ANCHOR` 做真正的“程序化正信号候选”准备

现在 `OA_TRUE_TARGET_LHS_AMPLIFICATION_MULTI_ANCHOR` 已经被明确提升为 `prepare_programmatic_positive_signal` 候选。  
这意味着下一步不再只是“承认它存在”，而是要开始设计：

- 如果将来不用改 prompt wording，而是通过更隐式的方法利用它，该怎么做？

这里最合适的做法不是立刻改主 prompt，而是先做离线准备，例如：

- 把它整理成更明确的程序化触发条件
- 输出命中样本清单和边界样本清单
- 设计后续可接入的 feature / signal 形式
- 评估它与其他已有正向资产是否重叠过多

换句话说，下一步不是“立刻注入”，而是“把注入前的准备做到足够清楚”。

### 22.4 第四优先级：只有在更大切分验证通过后，才考虑 `P1_2_3` 的轻注入实验

这一步仍然要非常克制。

当前不建议直接进入 prompt 改动，除非同时满足：

1. 至少有 `1~2` 个新子型在更大切分上仍然稳定
2. 它们与 `P1_2_3` 的主漏判区域确实显著相关
3. 它们能用非常短、非常弱、非常隐式的方式表达
4. 不会重新引入强 false-filter 或强决策层

如果这些条件不满足，就应该继续把成果保留在离线资产层，而不是为了“感觉上应该有帮助”去贸然改主 prompt。

### 22.5 第五优先级：继续修整测试与本地验证方式，保证工程迭代不被环境问题卡住

这属于工程支撑工作，优先级不如上面几条高，但也值得继续完善。

目前已经确认：

- 当前 Windows + `DLEnv` 环境中的 `pytest` 仍然会受到第三方插件加载和临时目录权限问题影响
- 因此近期的相关改动，是通过“直接执行测试函数”的方式完成验证的

这说明项目当前并不是“没有测试”，而是“测试逻辑本身可以跑，但测试宿主环境还需要继续清理”。  
后续如果有时间，仍应继续把：

- pytest tmpdir
- 权限清理
- 自动验证入口

整理得更稳定一些。  
这样做虽然不直接提高模型效果，但能让后续 family tagger / offline asset 迭代更省时间。

### 22.6 用一句话概括当前真正的下一步

接下来最应该做的，不是再去大范围改 prompt wording，而是：

**先把最新细切出来的结构子型拿到更大固定切分上验证，并继续完成剩余高价值主轴的人工复核；只有这些结果都稳定后，才考虑把极少数高置信子型转成更隐式的程序化正信号，最后再决定是否值得对 `P1_2_3` 做极小幅度轻注入。**
