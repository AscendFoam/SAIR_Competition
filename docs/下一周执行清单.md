# SAIR 代数推理竞赛下一周执行清单

截至日期：2026-04-05

## 1. 文档目的

这份清单不是泛泛的“接下来可以做什么”，而是把当前阶段最值得推进的事项，拆成一周内可执行、可检查、可交接的日程表。

本周的核心原则只有一句话：

**先把离线结构资产这条线做深，再决定是否对主线 prompt 做极小幅度注入；不要重新进入大规模 wording 试错。**

当前默认前提：

- 稳态主分支继续保持 `P1_2_3_implicit_guardrail_v2`
- 高召回挖掘支线继续保留 `P1_2_5`
- 主工作重心放在 `review set -> 人工复核 -> family tagger 扩展 -> 更大切分验证`

---

## 2. 本周目标

本周只追求四个结果：

1. 完成一轮高优先级 review set 人工复核
2. 把复核结论落实为更细的 family tagger / 结构标签
3. 在比 `smoke` 更大的固定切分上验证这些结构标签是否仍然成立
4. 基于验证结果，判断是否值得对 `P1_2_3` 做一次“轻注入”式小迭代

本周**不以新 prompt 分数上涨为第一目标**。  
本周第一目标是把“人类观察到的结构规律”变成“可重复验证的工程资产”。

## 3. 工作分工总览

## 3.1 需要你人工完成的部分

- 阅读并回填 `P0/P1 review template`
- 判断某个样本是否真的应该属于当前 axis
- 判断某些 boundary tags 是否值得升成新标签
- 给出更窄结构族的命名与自然语言解释
- 最终决定哪些规律可以进入主线，哪些只能保留为离线资产

这些事情不能完全自动化，因为它们本质上是“结构理解”和“规则命名”工作。

## 3.2 我可以帮你处理的部分

- 整理和改写 review 模板
- 扩展 `family_tagger.py` 及相关测试
- 增加新的 CLI / 统计脚本 / 报告脚本
- 批量跑数据、汇总标签分布、生成对比报告
- 根据你的人工复核结论，把标签沉淀成结构化资产
- 帮你设计“轻注入”版 prompt 的最小改动实验

## 3.3 可以交给脚本自动化的部分

- 打标签
- 附着标签到已有 predictions
- 构建 offline rule assets
- 轴合并与去重
- review set 构建
- 各 split 上的统计汇总
- prompt 候选对比报告

## 4. 每日执行清单

## Day 1：吃掉 P0/P1 人工复核

### 目标

把当前最有价值的 review rows 人工看完，并形成明确结论。

### 你需要做的事

1. 打开下面两个文件：
   - `reports/experiments/offline_rule_review_set_smoke/p0_p1_review_template.md`
   - `reports/experiments/offline_rule_review_set_smoke/annotation_checklist.md`
2. 优先只看这三类 axis：
   - `OA_TRUE_SINGLETON_WITH_TARGET_NEW_VARS`
   - `OA_TRUE_DISJOINT_BINARY_BINARY`
   - `OA_TRUE_SINGLETON_WITH_TARGET_SHARED_LHS`
3. 对每个样本至少回填这四项：
   - 当前 axis 是否正确
   - 是否应该拆出更窄 child axis
   - 哪些 boundary tags 值得保留
   - 这类规律更适合 `family tagger` 还是更适合以后做隐式注入
4. 把结论直接写在模板里，不需要写得很正式，但要明确。

### 我可以帮你做的事

- 如果你回填后把模板发给我，我可以帮你把结论整理成：
  - 新标签清单
  - family tagger 变更方案
  - 下一轮统计脚本输入

### 当天产出

- 一份已回填的 `p0_p1_review_template.md`
- 一份你自己的简短判断摘要，建议 5~10 行即可

### 完成标准

- `P0` 全部看完
- `P1` 至少看完 `OA_TRUE_SINGLETON_WITH_TARGET_SHARED_LHS`

## Day 2：把人工判断转成“标签设计稿”

### 目标

把 Day 1 的人工结论收敛成明确的新标签候选，而不是停留在描述性语言。

### 你需要做的事

1. 从 Day 1 的回填内容里提取：
   - 哪些结构模式反复出现
   - 哪些 boundary tags 只是噪声
   - 哪些样本其实属于同一个更窄结构族
2. 对每个候选新标签写三行说明：
   - 标签名
   - 结构定义
   - 预期用途

建议优先考虑的标签方向：

- `singleton + target new vars`
- `singleton + shared lhs`
- `disjoint binary-binary`
- `constant-operation candidate` 的主轴/边界拆分

### 我可以帮你做的事

- 把你的结论整理成正式标签提案
- 直接修改 `src/sair_competition/features/family_tagger.py`
- 补充对应单元测试
- 更新 tag 说明文本

### 当天产出

- 一份“新标签设计稿”

建议格式：

```text
tag_name:
structural definition:
example rows:
keep / split / drop reason:
expected use:
```

### 完成标准

- 至少产出 2~4 个可实现的新标签候选

## Day 4：把验证从 smoke 推到更大切分

### 我可以帮你做的事

  - 样本数
  - true/false 分布

### 可自动化执行的内容

- 与现有 predictions 对齐

### 希望看到的结论

- 哪些新标签仍然高纯度
- 哪些在大样本下其实不稳定
- 哪些只是 `smoke` 偶然现象

### 当天产出

- 一份“标签泛化性验证摘要”

### 完成标准

- 至少知道 2 类标签可以继续保留

## Day 6：决定是否做一次“轻注入”式 prompt 迭代

### 目标

只有在前五天的结构验证充分时，才考虑对主线 prompt 做一次非常小的注入实验。

### 判断条件

必须同时满足以下条件，才建议进入 prompt 改动：

1. 至少有 1~2 个窄结构标签在更大切分上仍然稳定
2. 它们与 `P1_2_3` 的主要漏判显著相关
3. 这些规律可以用很短的方式表达
4. 不需要引入强 false-filter 或强决策层

### 你需要做的事

- 最终拍板：这轮是否值得碰主线 prompt

### 我可以帮你做的事

- 如果值得，我来给出：
  - 一版最小改动的 prompt 草案
  - 与 `P1_2_3` 的差异说明
  - 预期影响和风险说明
- 如果不值得，我来把结论回写到报告中，标注为“继续保留离线资产，不进入 prompt”

### 当天产出

二选一：

- `轻注入版 prompt 草案`
- 或 `本轮不改 prompt 的决策说明`

### 完成标准

- 不允许“因为好像有点道理”就贸然开新的大分支

## 7. 本周推荐协作方式

最推荐的协作节奏是：

1. 你先做 `P0/P1` 模板人工回填
2. 你把回填后的内容发给我
3. 我根据你的结论改 tagger、补测试、跑统计
4. 我把新统计结果再反馈给你
5. 你据此决定是否进入 prompt 轻注入

也就是说：

- **人负责结构判断**
- **我负责工程落实和批量验证**

这样分工最省时间，也最不容易跑偏。
